Phase 1 — Deep Dive
📥 Storing Your Documents

This runs once (or whenever you update your documents). The result is a searchable vector database.

1
Load Documents
Read your raw text — PDFs, Word docs, web pages, databases. Strip formatting, keep content.
Input: "The Moon landing occurred on July 20, 1969. Neil Armstrong..."
"Einstein developed the theory of relativity in 1905..."
2
Chunk Into Pieces
Split long documents into small overlapping windows (~300 characters). This keeps each chunk focused on one idea — easier to retrieve precisely.
Chunk 1: "The Moon landing occurred on July 20, 1969. Neil Armstrong..."
Chunk 2: "Neil Armstrong became the first human to walk on..."
Chunk 3: "...walk on the Moon. Buzz Aldrin followed shortly after."
3
Embed Each Chunk
Run each chunk through a sentence-transformer model. The output is a 384-number vector (an "embedding") that encodes the meaning of the text.
"Moon landing July 1969" → [0.23, -0.41, 0.87, 0.12, ..., 0.05]
← 384 numbers representing the meaning →
4
Store in FAISS Vector Database
Save all vectors in FAISS — Facebook's fast similarity search library. Now your documents are searchable by meaning, not just keywords.
FAISS index: 1,247 vectors indexed
Storage: ~0.5 MB for 1,000 chunks
Search speed: <10ms for 1M vectors
Phase 2 — Deep Dive
🔍 Answering a Question

This runs every time a user asks a question. Fast, deterministic, no hallucination from retrieval.

1
Receive the Query
The user types a question in natural language. No preprocessing needed.
User query: "Who was the first person to walk on the Moon?"
2
Embed the Query
Run the same embedding model on the query. This gives us a vector in the same 384-dimensional space as all our chunks.
"first person walk Moon" → [0.19, -0.38, 0.91, 0.09, ..., 0.07]
3
Vector Similarity Search
FAISS finds the chunks whose vectors are closest to the query vector using L2 distance. "Closest" means "most similar in meaning".
Score 0.94: "Neil Armstrong became the first human to walk on..."
Score 0.81: "The Moon landing occurred on July 20, 1969..."
Score 0.67: "Buzz Aldrin was the second person to walk..."
4
Retrieve Top-K Chunks
Take the top 3 most relevant chunks. These become the "context" for the LLM. You can adjust K from 1–10 depending on how much context you need.
Retrieved 3 chunks from 1,247 indexed
Total context: ~900 characters
Retrieval time: 3ms
5
Generate Answer
The LLM reads the retrieved chunks + the question and generates an answer grounded in your documents. It cannot hallucinate what isn't there.
Prompt: "Using ONLY this context, answer: Who first walked on Moon?\nContext: Neil Armstrong became the first human..."
Answer: "Neil Armstrong was the first person to walk on the Moon."
Core Vocabulary
📚 Key Concepts Explained Simply

The five things you must understand to work with any RAG system.

🔢
Embedding
Text → a list of 384 numbers. Similar texts produce similar number lists. This is how computers understand "meaning".
✂️
Chunking
Splitting a document into ~300-character pieces with overlap (~50 chars). Overlap prevents context from being cut off at boundaries.
🗄️
FAISS
Facebook AI Similarity Search. A fast library for finding nearest neighbors in high-dimensional space. Searches 1M vectors in milliseconds.
📐
L2 Distance
How "far apart" two vectors are. Closer = more similar. Score = 1 / (1 + distance). Score 1.0 = identical, 0.0 = unrelated.
🔝
Top-K
Retrieve the K most similar chunks. K=3 is a common default — enough context without overwhelming the LLM.
🧠
Sentence Transformer
The embedding model (e.g., all-MiniLM-L6-v2, 22MB). Converts any text to 384-dim vectors that capture semantic meaning.
Intuition Builder
🎓 The Open Book Exam Analogy

❌ Closed Book Exam (No RAG)

The student (LLM) can only use what they memorized before the exam (training cutoff). They may guess wrong and sound confident. Outdated, no private info.

✅ Open Book Exam (With RAG)

Before answering, the student looks up the relevant page in their textbook (retrieval). Then they answer from what they read — grounded, accurate, verifiable.
