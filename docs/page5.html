Naive RAG
The Foundation — every other RAG builds on this
⭐ Simple ⚡ Fastest ›
Theory

The original, simplest RAG architecture. Converts everything to meaning-vectors (embeddings), then finds the closest match at query time using L2 distance in FAISS.

→

→

🗄️ FAISS
❓ Query
→
🔢 Embed
→

→
💬 Generate

Key limitation: Purely semantic — misses exact keyword matches like "NASA" vs "National Aeronautics".

Use When / Avoid When

✅ Use when: Simple Q&A, prototypes, learning, corpora where semantic search is sufficient.

❌ Avoid when: Need exact keyword matches (names, codes, product IDs), or very large corpora.

Code
from rag_core import NaiveRAG

rag = NaiveRAG(chunk_size=300, overlap=50)
rag.add_documents(["Your document text here..."])
result = rag.ask("Your question?")
print(result.answer)
print(f"Top match score: {result.retrieved_docs[0].score:.2f}")
Interview Q&A
Q: Explain Naive RAG and its limitations.
Naive RAG embeds documents into a vector space and retrieves the nearest neighbors to the query vector. It fails when queries use different vocabulary than the source (e.g., "auto" vs "car"), and can't distinguish between semantically similar but factually different passages.

Hybrid RAG
BM25 keyword search + Vector search + RRF fusion
⭐⭐ Medium ›
Theory

Runs two independent retrievals — BM25 (keyword frequency) and FAISS (semantic vectors) — then fuses the rankings using Reciprocal Rank Fusion (RRF). Each method finds different chunks; fusing them gives better recall.

❓ Query
→
📊 BM25
+
🔢 Vector
→
🔀 RRF
→
💬 Answer

RRF formula: score(d) = Σ 1 / (k + rank(d)) across all rankers. k=60 is a common constant.

Use When / Avoid When

✅ Use when: Queries mix technical terms (BM25 excels) with semantic meaning (vector excels). Good for product docs, code search.

❌ Avoid when: Purely semantic queries — overhead of BM25 adds latency without benefit.

Interview Q&A
Q: What is RRF and why is it used in Hybrid RAG?
RRF (Reciprocal Rank Fusion) merges two ranked lists into one by summing 1/(k+rank) for each document across all rankers. It's parameter-light, robust, and consistently outperforms score-based fusion when the two rankers use incomparable score scales (e.g., BM25 term frequency vs cosine similarity).

Conversational RAG
Multi-turn memory — remembers prior questions
⭐⭐ Medium 🧠 Memory ›
Theory

Maintains a sliding window of prior Q&A turns as conversation history. Each new query is enriched with context from the conversation before retrieval, so follow-up questions like "What else?" work correctly.

Use When / Avoid When

✅ Use when: Building chatbots where users ask follow-up questions, multi-turn customer support, tutoring systems.

❌ Avoid when: Each query is independent (no follow-ups needed) — history adds latency with no benefit.

Interview Q&A
Q: How does Conversational RAG handle context window limits?
It uses a sliding window — only the last N turns are kept in memory. Older turns are dropped or summarized. This prevents the context from growing unbounded while still enabling short-term follow-ups.

Rerank RAG
Fast retrieve top-20 → cross-encoder rerank to top-3
⭐⭐ Medium High Precision ›
Theory

Two-stage retrieval: Stage 1 uses fast vector search to get top-20 candidates. Stage 2 runs a slower but more accurate cross-encoder that reads query+document jointly to reorder the final top-3.

❓ Query
→
🔍 Top-20
→
⚖️ Reranker
→
✅ Top-3
→
💬 Answer

Key difference: Bi-encoders embed query/doc separately (fast). Cross-encoders read them together (slow but more accurate).

Use When / Avoid When

✅ Use when: High-stakes domains (legal, medical, finance) where precision matters more than speed.

❌ Avoid when: Real-time latency requirements under 10ms — the cross-encoder adds 15–30ms.

Contextual RAG
Token-efficient — compresses chunks before sending to LLM
⭐⭐ Medium ›
Theory

After retrieving chunks, a compression step extracts only the sentences relevant to the specific query. This reduces LLM input tokens by 60–80%, cutting cost and latency while often improving answer quality (less noise).

→

→

💬 Answer
Use When / Avoid When

✅ Use when: Long documents where chunks contain irrelevant filler, high token-cost LLMs, or when you need to fit more context into a limited window.

❌ Avoid when: Short chunks are already focused — compression overhead isn't worth it.

Self-Reflective RAG (SELF-RAG)
Evaluates its own retrieval quality at every step
⭐⭐⭐ Hard ›
Theory

Introduced by Asai et al. (2023). Adds reflection tokens at each step: before retrieval (should I retrieve?), during (is this chunk relevant?), and after (is my answer good?). Can decide NOT to retrieve if the LLM already knows the answer.

❓ Query
→
🤔 Reflect?
→
🔍 Retrieve
→
📊 Critique
→
✅ Answer
Reflection Tokens
  • [Retrieve] — should I look something up?
  • [IsRel] — is this chunk relevant?
  • [IsSup] — does the chunk support my answer?
  • [IsUse] — is this answer useful / confident?
Interview Q&A
Q: How does SELF-RAG differ from standard RAG?
Standard RAG always retrieves and always uses what it retrieves. SELF-RAG can skip retrieval if the model is already confident, reject irrelevant chunks mid-pipeline, and flag low-confidence answers — all via learned reflection tokens inserted into the generation stream.

✅
Corrective RAG (CRAG)
Grades every retrieved chunk and corrects bad retrievals
⭐⭐⭐ Hard ›
Theory

Every retrieved chunk gets a grade: CORRECT (confident, use it), AMBIGUOUS (partially useful, compress it), or INCORRECT (wrong, discard and web-search instead).

🔍 Retrieve
→
📊 Grade
→
✅ Use
or
🌐 Search
Interview Q&A
Q: What happens when CRAG grades a chunk as INCORRECT?
CRAG discards the chunk and falls back to a web search (or a backup retrieval source) to find better evidence. This makes it particularly robust for queries where the local knowledge base doesn't contain the answer.

Adaptive RAG
Classifies query complexity and routes to the right RAG
⭐⭐⭐ Hard ›
Theory

An upstream classifier analyzes each query and routes it: simple factual questions → Naive RAG (fast), keyword-heavy → Hybrid RAG, complex comparative → Rerank RAG. Optimal for production systems with mixed workloads.

❓ Query
→
🧠 Classifier
→
🔵 Naive
/ 🟣 Hybrid / 🔶 Rerank
Routing Rules
  • Short / date queries → Naive RAG
  • Technical terms / acronyms → Hybrid RAG
  • "Compare", "contrast", "best" → Rerank RAG
  • Multi-part ("and", "also") → Multi-Hop RAG

Multi-Hop RAG
Chains retrievals — each hop finds new entities to look up
⭐⭐⭐ Hard ›
Theory

For questions that require bridging multiple facts (e.g., "Who invented the technology used in the satellite that first reached Mars?"), Multi-Hop extracts entities from each retrieved chunk and uses them as queries for the next hop.

Interview Q&A
Q: When does Multi-Hop RAG outperform Naive RAG?
When the answer requires connecting two separate facts that don't co-occur in any single chunk. Naive RAG finds one relevant passage; Multi-Hop uses the entities in that passage as a stepping stone to a second, different passage, then synthesizes both.

Graph RAG
Knowledge graph traversal for relationship-heavy data
⭐⭐⭐ Hard 📚 Documented ›
Theory

Builds a knowledge graph (entities as nodes, relationships as edges) from documents. Query traversal follows entity links rather than vector similarity, enabling answers that depend on graph structure (e.g., "What connects X and Y?").

Agentic RAG
An AI agent plans and executes its own retrieval strategy
⭐⭐⭐ Hard 📚 Documented ›
Theory

An LLM agent is given a set of retrieval tools and autonomously decides what to search, when to stop, and how to synthesize. It can search multiple sources, generate sub-queries, and reason step-by-step before producing a final answer.

Research automation, complex multi-source synthesis, open-ended questions with no fixed retrieval path. Requires a capable LLM (GPT-4 class or better) to reason about tool selection reliably.
